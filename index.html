Pinelabs SDK Design principle · v0.1.0
Kotlin · Quickstart 
// 1 — Init once per process
val sdk = PinelabsSdk.init(
    context,
    appId       = "1001",
    userId      = "cashier-01",
    version     = "2.0",
    timeoutSecs = 120
)

// 2 — Dispatch a sale (₹99.00 = 9900 paise)
sdk.doTransaction(
    TransactionRequest(
        transactionType = TransactionType.SALE,
        amount          = 9900uL,
        billingRefNo    = "ORD-20260429",
        invoiceNo       = "INV-001"
    ),
    object : TransactionListener {
        override fun onStarted(eventId: String) { /* ack */ }
        override fun onSuccess(r: TransactionResult) { /* approved */ }
        override fun onFailure(e: SdkException) { /* handle */ }
    }
)
01

Getting Started

The Pinelabs Common SDK lets an Android merchant application drive a Pine Labs POS terminal via app-to-app integration with the on-device MasterApp (com.pinelabs.masterapp). The SDK abstracts the IPC (Messenger + JSON payload) and exposes a small, strongly-typed Kotlin surface.

💡

Minimum end-to-end flow

  1. Call init once per process with the merchant identity and timeout.
  2. Build a TransactionRequest.
  3. Call doTransaction with a TransactionListener.
  4. Handle onStarted (acknowledgement with eventId), onSuccess, or onFailure.

02

Conventions

💰
Amounts are always in the smallest currency unit (paise for INR). 9900 = ₹99.00.
🔑
eventId is a UUIDv4 string generated by the SDK on each doTransaction. Callers never supply one.
🕐
Timestamps on TransactionResult are reported as date (MMDDYYYY) and time (HHmmss) — not ISO‑8601.
🧵
Thread model: listener callbacks are dispatched on pinelabs-sdk-worker. Never perform UI work inside a callback.
🔄
Concurrency: only one operation is in flight per SDK instance. doTransaction calls are serialized on the worker thread.
♻️
Initialization: init is idempotent — repeated calls return the existing instance.

TransactionListener

All three callbacks run on the SDK worker thread. Exactly one terminal callback (onSuccess or onFailure) fires per doTransaction invocation. onStarted always precedes the terminal callback.

Kotlin 
interface TransactionListener {
    fun onStarted(eventId: String)
    fun onSuccess(result: TransactionResult)
    fun onFailure(error: SdkException)
}

Error Reference

All errors surface through TransactionListener.onFailure(error: SdkException). The base type is a sealed set; pattern-match on the subclass.

SdkException

VariantFieldsRecoverableMeaning
NotInitialized—NodoTransaction was called before init.
InvalidInputdetail: StringNoRequest validation failed (e.g. zero amount, blank reference).
TransportUnavailabledetail: StringYesMasterApp missing, bind failed, or IPC channel broken. Retry after verifying MasterApp is running.
Timeoutdetail: StringYesNo reply within timeoutSecs. Retry; consider a longer timeout for slow networks.
TransactionFaileddetail: String, terminalResponseCode: String?dependsTerminal returned a non-zero response code. detail carries ResponseMsg.
Internaldetail: StringNoUnexpected condition (e.g. malformed response JSON). Report to Pinelabs with eventId.

Common terminal response codes

CodeMeaning
"0"Approved — delivered as onSuccess, not onFailure.

Troubleshooting

🔗 UnsatisfiedLinkError: libuniffi_pinelabs.so not found
Cause: ABI filter missing or wrong AAR version.
Fix: Add ndk { abiFilters 'arm64-v8a', 'armeabi-v7a', 'x86_64' }; confirm AAR is 0.1.0.
📡 TransportUnavailable("MasterApp service not available")
Cause: MasterApp not installed or <queries> stripped from host manifest.
Fix: Install MasterApp; remove tools:node="remove" on any <queries> element.
⏱️ Timeout on every transaction
Cause: Terminal busy, PED stall, or timeoutSecs too low.
Fix: Retry; raise timeoutSecs; check terminal home screen.
🔒 TransactionFailed(detail="Batch is Locked…", terminalResponseCode="7")
Cause: Settlement pending.
Fix: Settle batch on terminal, then retry.
🏦 TransactionFailed with other codes
Cause: Real acquirer decline.
Fix: Surface responseMessage to the user.
