Concepts · v1

event_id & Reconciliation

Every transaction the SDK handles is identified by an event_id — a UUIDv4 allocated by the SDK on doTransaction and handed back via TransactionListener.onStarted. This page explains what that ID is for and exactly how to use it for crash-recovery and bookkeeping.

01

Who allocates the event_id

The SDK. Always. You never construct it, you never guess it, you do not get to pick the format. onStarted fires before onSuccess / onFailure iff the SDK accepted the call.

⚠️
No event_id when the call fails synchronously

If doTransaction raises SdkException synchronously (e.g. InvalidInput, TransportUnavailable during bind), no event_id is allocated and no callback fires.

Kotlin 
sdk.doTransaction(request, object : TransactionListener {
    override fun onStarted(eventId: String) {
        // SDK allocated a UUIDv4 for this operation.
        // Persist it NOW (see §03) before showing any UI.
        db.insertOpInFlight(orderId = currentOrderId, eventId = eventId)
    }
    override fun onSuccess(result: TransactionResult) { /* … */ }
    override fun onFailure(error: SdkException) { /* … */ }
})
Java 
sdk.doTransaction(request, new TransactionListener() {
    @Override public void onStarted(String eventId) {
        // SDK allocated a UUIDv4 for this operation.
        // Persist it NOW (see §03) before showing any UI.
        db.insertOpInFlight(currentOrderId, eventId);
    }
    @Override public void onSuccess(TransactionResult result) { /* … */ }
    @Override public void onFailure(SdkException error) { /* … */ }
});
02

Lifecycle states

Each event_id passes through one of these states. Query the current state at any time with checkStatus(eventId, options).

Pending
SDK accepted, transport not yet engaged
InFlight
Transport engaged, awaiting terminal reply
success
Completed
failure
Failed
cancel
Cancelled
Unknown
SDK has no record of this id — process restarted, never allocated, or registry purged. See §04.
Kotlin 
val status = sdk.checkStatus(eventId, options)
when (status.state) {
    OperationState.Pending,
    OperationState.InFlight   -> showSpinner()
    OperationState.Completed  -> renderResult(status.result!!)
    OperationState.Failed     -> showError(status.failureDetail, status.terminalResponseCode)
    OperationState.Cancelled  -> showCancelled()
    OperationState.Unknown    -> handleUnknown(eventId)   // see §04
}
Java 
OperationStatus status = sdk.checkStatus(eventId, options);
switch (status.getState()) {
    case Pending:
    case InFlight:   showSpinner(); break;
    case Completed:  renderResult(status.getResult()); break;
    case Failed:     showError(status.getFailureDetail(),
                               status.getTerminalResponseCode()); break;
    case Cancelled:  showCancelled(); break;
    case Unknown:    handleUnknown(eventId); break;   // see §04
}
03

The persistence contract

This is the rule that keeps you reconciled:

📌
Persist event_id to durable storage in onStarted — before any UI update

Also persist the terminal-state outcome together with the same ID in onSuccess / onFailure. If your app crashes between onStarted and the terminal callback, the event_id is the only thing that lets you discover what really happened — the customer's card may have been charged.

Kotlin 
val listener = object : TransactionListener {

    override fun onStarted(eventId: String) {
        // ✅ Persist BEFORE updating UI
        db.insertOpInFlight(
            orderId    = currentOrderId,
            eventId    = eventId,
            startedAt  = now()
        )
        runOnUiThread { disablePayButton() }
    }

    override fun onSuccess(result: TransactionResult) {
        // ✅ Mark terminal state with outcome
        db.markOpCompleted(
            eventId = result.eventId,
            txnId   = result.transactionId
        )
        runOnUiThread { showSuccess(result) }
    }

    override fun onFailure(error: SdkException) {
        // ✅ Persist failure detail for back-office
        db.markOpFailed(
            eventId = currentEventId,
            error   = error.toString()
        )
        runOnUiThread {
            when (error) {
                is SdkException.TransactionFailed    -> showDecline(error.detail, error.terminalResponseCode)
                is SdkException.Timeout,
                is SdkException.TransportUnavailable -> showAmbiguous()   // call checkStatus!
                is SdkException.InvalidInput         -> log.error("Programmer error: ${error.detail}")
                is SdkException.Internal             -> log.error("SDK error: ${error.detail}")
                is SdkException.NotInitialized       -> log.error("SDK not initialised")
            }
        }
    }
}
Java 
TransactionListener listener = new TransactionListener() {

    @Override
    public void onStarted(String eventId) {
        // ✅ Persist BEFORE updating UI
        db.insertOpInFlight(currentOrderId, eventId, now());
        runOnUiThread(() -> disablePayButton());
    }

    @Override
    public void onSuccess(TransactionResult result) {
        // ✅ Mark terminal state with outcome
        db.markOpCompleted(result.getEventId(), result.getTransactionId());
        runOnUiThread(() -> showSuccess(result));
    }

    @Override
    public void onFailure(SdkException error) {
        // ✅ Persist failure detail for back-office
        db.markOpFailed(currentEventId, error.toString());
        runOnUiThread(() -> {
            if (error instanceof SdkException.TransactionFailed e) {
                showDecline(e.getDetail(), e.getTerminalResponseCode());
            } else if (error instanceof SdkException.Timeout
                    || error instanceof SdkException.TransportUnavailable) {
                showAmbiguous();   // call checkStatus!
            } else if (error instanceof SdkException.InvalidInput e) {
                log.error("Programmer error: " + e.getDetail());
            } else if (error instanceof SdkException.Internal e) {
                log.error("SDK error: " + e.getDetail());
            } else if (error instanceof SdkException.NotInitialized) {
                log.error("SDK not initialised");
            }
        });
    }
};
04

OperationState.Unknown — what to do

Unknown is returned in three legitimate situations. It is not an error from the SDK's view — checkStatus returns it cleanly without raising InvalidInput.

Situation Recovery
SDK was restarted The in-memory registry was wiped. The op may have completed at the terminal before the crash.

App-to-App / PADController / TCP: irrecoverable via SDK — reconcile via Pinelabs back-office reports keyed by billingRefNo / invoiceNo / referenceId.

Cloud: call checkStatus(eventId) again — the cloud server is the source of truth, not the SDK's in-memory registry.
ID from a different SDK instance Two SDK instances ran in the same process; the ID came from the other one. Consolidate to a single SDK instance per the lifecycle guide.
Memory pressure / explicit purge Long-lived processes may purge completed ops from the registry over time. Treat identically to the restart case above.
05

event_id semantics by transport

The ID you pass to cancel and checkStatus differs by transport:

Transport What event_id means for cancel / checkStatus
App-to-App The SDK-allocated UUIDv4 from onStarted. Both cancel and checkStatus raise NotSupported — the protocol does not expose them.
Cloud The PlutusTransactionReferenceID returned by Upload, not the SDK UUIDv4. The SDK mirrors the cloud reference into TransactionResult.transaction_id on the Pending callback — pass that value to cancel / checkStatus. Persisting the transaction_id from the Cloud Pending callback is just as critical as persisting the event_id.
PADController The SDK-allocated UUIDv4.
TCP The SDK-allocated UUIDv4.

This asymmetry exists because the cloud server has its own opaque identifier that the SDK cannot translate back into a UUIDv4 after a restart.

06

Idempotency

The SDK does not auto-retry. If onFailure(Timeout) fires, your app decides what to do based on retryability:

✓ Retryable — re-issue doTransaction
  • TransportUnavailable
  • Timeout
  • NotConnected
  • TransportError
✗ Terminal — do not re-issue
  • TransactionFailed
  • InvalidInput
  • NotSupported
  • OperationInProgress
  • Cancelled
  • Internal

When you re-issue, each attempt gets a new event_id. Some integrators reuse the merchant referenceId / billingRefNo across attempts so back-office reconciliation can identify them as one logical order.

☁️
Cloud server-side idempotency key

CloudTransactionOptions.transactionNumber is the Pinelabs server-side idempotency key. Re-uploading with the same transactionNumber returns the original Pinelabs reference without re-charging the card — use this if your retry is within the window the cloud server still remembers.

07

cancel semantics

cancel(eventId, options) is best-effort:

✅
Cancel succeeded — if the transport supports a server-side cancel and the terminal has not yet committed, the eventual outcome moves to OperationState.Cancelled.
⚡
Cancel is a no-op — if the terminal has already committed (the card was charged), cancel does nothing. The outcome remains Completed / Success.
⚠️
Always wait for the terminal-state callback

Treat cancel as advisory. Your code must wait for onSuccess / onFailure or poll checkStatus to confirm the final outcome. The SDK never silently auto-cancels.

08

Practical reconciliation workflow

The happy path and the crash-recovery path, step by step:

Happy path

  1. 1
    User taps "Charge ₹X". App inserts a pending order row with merchant order ID and shows a spinner.
  2. 2
    App calls sdk.doTransaction(req, listener). If a synchronous exception is thrown, show an error and stop.
  3. 3
    listener.onStarted(eventId) fires → update the order row with event_id and startedAt. Persist to durable storage before touching the UI.
  4. 4
    onSuccess(result) or onFailure(error) fires → update the row with the terminal-state outcome; print receipt or show error; close.

Crash recovery (on app restart)

  1. 1
    On startup, fetch all order rows with state Pending or InFlight.
  2. 2
    App-to-App / PADController / TCP: the in-memory registry was wiped — these IDs are irrecoverable via the SDK. Mark them "needs back-office reconciliation" and resolve via Pinelabs reports keyed by billingRefNo / invoiceNo.
  3. 3
    Cloud: read the persisted transaction_id (cloud ref) from the row; call sdk.checkStatus(transactionId, CheckStatusOptions.Cloud(…)); update the row from the result.
Kotlin — startup reconciliation 
// Called once at app startup
fun reconcilePendingOps(sdk: PinelabsSdk) {
    val stuckRows = db.queryOpsInFlight()  // Pending or InFlight rows

    for (row in stuckRows) {
        when (row.transport) {
            Transport.CLOUD -> {
                // Cloud server is the source of truth
                val cloudRef = row.transactionId
                    ?: continue.also { markNeedsManualReconciliation(row) }

                val status = sdk.checkStatus(
                    eventId = cloudRef,
                    options = CheckStatusOptions.Cloud(/* … */)
                )
                when (status.state) {
                    OperationState.Completed -> db.markOpCompleted(row, status.result!!)
                    OperationState.Failed    -> db.markOpFailed(row, status.failureDetail)
                    OperationState.Cancelled -> db.markOpCancelled(row)
                    else                     -> markNeedsManualReconciliation(row)
                }
            }
            else -> {
                // App-to-App / PADController / TCP: irrecoverable via SDK
                markNeedsManualReconciliation(row)
            }
        }
    }
}
Java — startup reconciliation 
// Called once at app startup
void reconcilePendingOps(PinelabsSdk sdk) {
    List<OpRow> stuckRows = db.queryOpsInFlight();  // Pending or InFlight rows

    for (OpRow row : stuckRows) {
        if (row.transport == Transport.CLOUD) {
            // Cloud server is the source of truth
            String cloudRef = row.transactionId;
            if (cloudRef == null) {
                markNeedsManualReconciliation(row);
                continue;
            }
            OperationStatus status = sdk.checkStatus(
                cloudRef,
                new CheckStatusOptions.Cloud(/* … */)
            );
            switch (status.getState()) {
                case Completed: db.markOpCompleted(row, status.getResult()); break;
                case Failed:    db.markOpFailed(row, status.getFailureDetail()); break;
                case Cancelled: db.markOpCancelled(row); break;
                default:        markNeedsManualReconciliation(row); break;
            }
        } else {
            // App-to-App / PADController / TCP: irrecoverable via SDK
            markNeedsManualReconciliation(row);
        }
    }
}
