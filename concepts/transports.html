Transports

A transport is the channel between the SDK (running inside your app) and the Pinelabs payment terminal. v1 ships four transports — each with different prerequisites, latency characteristics, and capability footprints. 

┌────────────────────┐     ┌───────────────────┐
│ Your app + SDK     │     │ Pinelabs terminal │
└──────────┬─────────┘     └─────────┬─────────┘
           │                         │
           │  one of …               │
           │                         │
   ┌───────┴────┐  ┌──────────┐  ┌───┴─────────┐  ┌───────────────┐
   │ App-to-App │  │ Cloud    │  │PADController │  │ TCP direct    │
   │ (Android   │  │ REST     │  │(BT/USB/Ser.) │  │ (LAN; v1     │

02

Transport details

Each transport below covers its use-case, how it works internally, prerequisites, capability footprint, and threading model.

📱
App-to-App
Android IPC to MasterApp — same device
✓ Shipping
Use when
Your app runs on a Pinelabs all-in-one Android device (or any Android terminal where MasterApp is the system-side payment host).
How it works

The SDK binds to the MasterApp Messenger service (com.pinelabs.masterapp.SERVER), sends a JSON envelope as a Bundle, awaits a single reply, and unbinds. You receive a typed TransactionResult — you never see the JSON, the Bundle, or the IPC.

Prerequisites
  • MasterApp must be installed on the device (Pinelabs ships it on every in-store terminal). If absent, calls fail with SdkException.TransportUnavailable — the detail mentions "MasterApp not installed".
  • Your AndroidManifest.xml must declare the package query so Android 11+ doesn't filter it out:
    XML
    <queries>
    <package android:name="com.pinelabs.masterapp" />
</queries>
  • SdkConfig.applicationId must be set to the Pinelabs-provisioned identifier for your installing package. The MasterService validates it server-side for MethodId="1001" Sale; it bypasses validation for MethodId="1002" test-print.
  • SdkConfig.appToApp must carry userId and version = "1.0" (the production MASTERAPP_API_VERSION). Other values are rejected by MasterService.
Capability footprint
doTransaction testPrint cancel discoverTerminals diagnostics

connect / disconnect are accepted as no-ops. isConnected() returns true only while a call is in flight.

Threading

The SDK binds the service per call on a worker thread — bind / send / await / unbind happen synchronously inside one doTransaction. There is no long-lived connection.

☁️
Cloud REST
HTTPS upload to Pinelabs cloud — any platform
✓ Shipping (Phase 7a)
Use when
Your app runs anywhere with HTTPS egress and the terminal is reachable from the Pinelabs cloud. Typical for multi-store retail back-offices, call-center pay-by-link, and headless billing services.
How it works

The SDK posts an UploadBilledTransaction body to the configured cloudBaseUrl. On ResponseCode == 0 the listener fires onSuccess(TransactionResult { status = Pending, transactionId = PlutusTransactionReferenceID, … }). The merchant then drives the rest of the lifecycle via checkStatus(eventId, CheckStatusOptions.Cloud(merchantStorePosCode)) until the state moves to Completed or Failed.

⚠️
No auto-poll in v1

The SDK does not automatically poll checkStatus. Your app is responsible for polling until terminal state resolves.

Prerequisites
  • SdkConfig.transport = Cloud and SdkConfig.cloudBaseUrl = "https://<env>.pinelabs.example" — no trailing slash; one URL per environment, supplied by Pinelabs ops at integration time.
  • cloudConnectTimeoutMs (default 30 s) and cloudReadTimeoutMs (default 60 s) — tune for your network.
  • Every doTransaction requires transportOptions = TransportOptions.Cloud(transactionNumber, sequenceNumber, …). Both fields are required — the Pinelabs server uses them as part of its idempotency key.
  • Every cancel requires cancelOptions = CancelOptions.Cloud(amount).
  • Every checkStatus requires options = CheckStatusOptions.Cloud(merchantStorePosCode).
⚠️
allowedPaymentModes ignored in v1

The typed-enum-to-cloud-code mapping is not yet published. Pass the opaque code via CloudTransactionOptions.allowedPaymentMode (commonly "10").

Capability footprint
doTransaction cancel checkStatus testPrint discoverTerminals connect / disconnect diagnostics
Threading

Every Cloud call is a synchronous blocking HTTP exchange bounded by the configured timeouts. Call from a worker thread or wrap in your language-idiomatic async primitive (e.g. Kotlin coroutine, Java ExecutorService).

🖥️
PADController
TCP loopback to local PADController service — BT / USB / Serial
✓ Shipping (Phase 7c)
Use when
The terminal is physically attached to or paired with the host machine — Bluetooth headless terminal, USB-tethered PIN pad, or RS-232 cable in a back-office.
How it works

The SDK frames a CSV payload with the PADController binary header (Source ID 0x1000, FuncCode 0x0997 / 0x1997, length, body, 0xFF terminator), opens a TCP loopback to the local PADController process, sends the frame, receives the reply, parses the CSV, and returns a TransactionResult. Frame limit: 2048 bytes.

🔒
Wire format is not a public API

The CSV frame format is documented internally for SDK maintainers. It is never exposed to merchant code.

Prerequisites
  • PADController binary must be running on the host as a local service.
  • For Bluetooth: the terminal must be paired at the OS level before discoverTerminals() returns it.
  • For USB / Serial: the device path or COM port must be reachable; OS-level drivers must be installed.
Capability footprint

The broadest in v1:

doTransaction cancel checkStatus testPrint discoverTerminals connect / disconnect ping runSelfTest getLogs getTerminalInfo
Threading

PADController opens a one-shot TCP loopback connection per transaction. connect() validates that PADController is reachable; isConnected() reflects that liveness.

🔌
TCP direct
LAN-direct to terminal — placeholder only
⏳ Placeholder
🚫
Do not plan a v1 integration on TCP direct

The wire protocol is greenfield and will be designed at implementation time. Calls to setTransport(Tcp) in v1 raise SdkException.NotSupported. Use App-to-App, Cloud REST, or PADController.

The capability matrix includes TCP rows for forward compatibility only. All TCP capabilities currently return NotSupported.

🔄
Runtime switching is supported

You can switch transports at runtime with setTransport(…) — the SDK disconnects the current transport (if any) before swapping. Useful when you want App-to-App on a Pinelabs device and Cloud when running the same code on a tablet.

04

Switching transports

Configure all transports you may use at construction time. Call setTransport(…) on a worker thread when you want to swap. setTransport raises SdkException.InvalidInput if the target transport requires config you did not supply at construction (e.g. AppToApp without appToApp, or Cloud without cloudBaseUrl).

Kotlin
val sdk = PineBillingSdk(
    config = SdkConfig(
        defaultTimeoutMs = 60_000,
        transport = TransportType.AppToApp,
        appToApp = AppToAppConfig(
            userId  = "POS-42",
            version = "1.0",
        ),
        applicationId = "MERCHANT_PROVISIONED_ID",
        cloudBaseUrl  = "https://uat.pinelabs.example",
    ),
)

// Later, on a worker thread — swap to Cloud
sdk.setTransport(TransportType.Cloud)  // cloudBaseUrl already configured

sdk.doTransaction(
    request = TransactionRequest(
        transactionType = TransactionType.Sale,
        invoiceNo       = "INV-001",
        billingRefNo    = "BILL-001",
        amount          = 9900uL,
        transportOptions = TransportOptions.Cloud(
            transactionNumber = 1,
            sequenceNumber    = 1,
        ),
    ),
    listener = myListener,
)
Java
PineBillingSdk sdk = new PineBillingSdk(
    new SdkConfig.Builder()
        .defaultTimeoutMs(60_000)
        .transport(TransportType.APP_TO_APP)
        .appToApp(new AppToAppConfig("POS-42", "1.0"))
        .applicationId("MERCHANT_PROVISIONED_ID")
        .cloudBaseUrl("https://uat.pinelabs.example")
        .build()
);

// Later, on a worker thread — swap to Cloud
sdk.setTransport(TransportType.CLOUD);  // cloudBaseUrl already configured

TransactionRequest request = new TransactionRequest.Builder()
    .transactionType(TransactionType.SALE)
    .invoiceNo("INV-001")
    .billingRefNo("BILL-001")
    .amount(9900L)
    .transportOptions(new TransportOptions.Cloud(1, 1))
    .build();

sdk.doTransaction(request, myListener);
