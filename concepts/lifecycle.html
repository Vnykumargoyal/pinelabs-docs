Lifecycle & Threading

The contract for when to construct the SDK, how many instances to keep, and which thread your code runs on at every step. Get this wrong and you will see OperationInProgress errors, frozen UIs, or double-charged customers.

OperationInProgress
Frozen UI thread
Double-charge risk
01

Construction

pseudo-code 
sdk = PineBillingSdk(config, appToAppBridge)
Rule Why it matters
One instance per process The SDK keeps an in-memory operation registry keyed by event_id. Two instances → two registries → checkStatus returns Unknown for ops the other instance ran.
Construction is fast & synchronous No I/O is performed during construction. Safe to call from the main thread.
Config is validated at construction Invalid SdkConfig raises SdkError.InvalidInput immediately — catch misconfiguration at app launch, not at the first transaction.
appToAppBridge required for App-to-App Required iff you ever call setTransport(AppToApp). On Android the binding constructs the bridge automatically when you pass SdkConfig.appToApp.
Not enforced-idempotent Calling the constructor twice will not crash, but you will leak an SDK and confuse checkStatus. Some test harnesses need two SDKs against fakes — that is the only legitimate case.
🔒
Single-instance pattern — per language

Construct once at app startup and share the reference everywhere.

Kotlin
object SdkHolder { val sdk = PineBillingSdk(config, bridge) }
Java
@Provides @Singleton PineBillingSdk provideSdk() { return new PineBillingSdk(config, bridge); }
Swift
class SdkManager { static let shared = PineBillingSdk(config: cfg, bridge: bridge) }
Python
_sdk = PineBillingSdk(config, bridge) # module-level, imported once
Node.js
export const sdk = new PineBillingSdk(config, bridge); // default export
C
static pine_billing_sdk_t *g_sdk; pthread_once(&once, sdk_init);
Kotlin — Application.onCreate 
class MyApplication : Application() {
    lateinit var sdk: PineBillingSdk

    override fun onCreate() {
        super.onCreate()
        val config = SdkConfig(
            appToApp = AppToAppConfig(context = this),
            logLevel  = LogLevel.INFO
        )
        sdk = try {
            PineBillingSdk(config, AppToAppBridge(this))
        } catch (e: SdkException.InvalidInput) {
            Log.e(TAG, "SDK misconfiguration: ${e.detail}")
            throw e   // crash-fast: no point starting with bad config
        }
    }
}
Java — Application.onCreate 
public class MyApplication extends Application {
    private PineBillingSdk sdk;

    @Override
    public void onCreate() {
        super.onCreate();
        SdkConfig config = new SdkConfig.Builder()
            .appToApp(new AppToAppConfig(this))
            .logLevel(LogLevel.INFO)
            .build();
        try {
            sdk = new PineBillingSdk(config, new AppToAppBridge(this));
        } catch (SdkException.InvalidInput e) {
            Log.e(TAG, "SDK misconfiguration: " + e.getDetail());
            throw e;  // crash-fast: no point starting with bad config
        }
    }

    public PineBillingSdk getSdk() { return sdk; }
}

runOnUiThread { … } · DispatchQueue.main.async { … } · Handler(mainLooper).post { … }
Caller thread
Your code
  • Listener methods return immediately — safe on any thread including UI.
  • Blocking methods perform I/O inline — never call from the UI thread.
  • No blocking I/O is done on the caller's stack during a listener call.
SDK worker thread
Rust / Tokio pool
  • Owned by the Rust core — not configurable.
  • All transport I/O: HTTPS, TCP, PAD framing, bindService.
  • On Android, App-to-App spins a one-shot HandlerThread per call, created and quit in a single roundTrip().
Callback thread
SDK-internal
  • All callbacks fire here — serialised, never concurrent on the same listener.
  • Never fired on the caller's stack frame before the call returns.
  • Must not block — hand off to UI dispatcher and return.
  • Must not re-enter SDK with listener-based methods synchronously.
✓ Listener-based — safe on any thread
  • doTransaction()
  • testPrint()
  • discoverTerminals()
✗ Blocking — must run on worker thread
  • cancel()
  • checkStatus()
  • connect() / disconnect()
  • ping() / runSelfTest()
  • getLogs() / getTerminalInfo()
  • setTransport()
⚠️
Android: SDK throws on main-thread blocking calls

The SDK detects Looper.getMainLooper() and throws IllegalStateException if you call a blocking method from the UI thread. Use Dispatchers.IO, an Executor, or WorkManager.

💡
Why the two-step model?

Listener-based methods can take tens of seconds at the terminal — cardholder inserting a card, entering a PIN, acquirer responding. A synchronous API would either block your UI for that time or force a thread-per-transaction model. The two-step model lets you write natural code: show a "tap card" UI, then update the screen when the callback fires.

Kotlin — blocking methods on worker thread 
// ✅ Blocking methods on a coroutine worker dispatcher
viewModelScope.launch(Dispatchers.IO) {
    val status = sdk.checkStatus(eventId, CheckStatusOptions.Cloud(/* … */))
    withContext(Dispatchers.Main) {
        updateUi(status)
    }
}

// ✅ connect() before first transaction
viewModelScope.launch(Dispatchers.IO) {
    try {
        sdk.connect(ConnectOptions.Tcp(host = "192.168.1.50", port = 9100))
    } catch (e: SdkException) {
        Log.e(TAG, "Connect failed: $e")
    }
}
Java — blocking methods on worker thread 
// ✅ Blocking methods on an Executor
ExecutorService io = Executors.newSingleThreadExecutor();

io.execute(() -> {
    try {
        OperationStatus status = sdk.checkStatus(eventId,
                new CheckStatusOptions.Cloud(/* … */));
        runOnUiThread(() -> updateUi(status));
    } catch (SdkException e) {
        Log.e(TAG, "checkStatus failed: " + e);
    }
});

// ✅ connect() before first transaction
io.execute(() -> {
    try {
        sdk.connect(new ConnectOptions.Tcp("192.168.1.50", 9100));
    } catch (SdkException e) {
        Log.e(TAG, "Connect failed: " + e);
    }
});
03

Concurrency: one operation per instance

The SDK enforces a single in-flight operation per PineBillingSdk instance:

Kotlin 
sdk.doTransaction(req1, listener1)   // ✅ accepted — onStarted fires

sdk.doTransaction(req2, listener2)   // ❌ throws SdkException.OperationInProgress
                                     //    .activeEventId = "<uuid of req1>"
Java 
sdk.doTransaction(req1, listener1);   // ✅ accepted — onStarted fires

try {
    sdk.doTransaction(req2, listener2);
} catch (SdkException.OperationInProgress e) {
    // ❌ throws — carry the active event ID to show "still busy with order #123…"
    showBusyMessage(e.getActiveEventId());
}
🔘
Disable the trigger control on onStarted — re-enable on onSuccess / onFailure

This is the canonical way to prevent double-charge bugs. The OperationInProgress exception carries the active_event_id so your UI can show "still processing order #123…" if it somehow fires.

cancel and checkStatus are not counted as operations — you can poll checkStatus(activeEventId) while a transaction is in flight to drive a "still waiting…" spinner on Cloud transport.

⚠️
Multiple SDK instances are not a concurrency workaround

The one-instance-per-process rule (§01) takes precedence. For throughput, serialise on a worker queue — one transaction at a time per terminal is the correct model.

04

Lifecycle of a single operation

From call site to terminal callback — what happens on each thread at each moment:

T0
Caller thread
sdk.doTransaction(req, listener)
SDK validates config, allocates a UUIDv4 event_id, enqueues work on the worker pool, and returns immediately. Your caller thread is free.
OperationState.Pending
T1
SDK worker thread
listener.onStarted(eventId)
The SDK fires onStarted — persist the event_id to durable storage now. The SDK then begins the active transport: bind service, open TCP, POST to cloud.
OperationState.InFlight
T2
SDK worker thread + Terminal
Terminal interaction
Cardholder taps / inserts card, enters PIN. Acquirer round-trip completes. This can take seconds to tens of seconds. Your UI thread remains responsive.
OperationState.InFlight
T3
SDK worker thread
listener.onSuccess(result) or onFailure(error)
Terminal state is settled. Persist the outcome, update your UI (via runOnUiThread), and re-enable the payment trigger control.
Completed · Failed · Cancelled
Query state between T0 and T3 — or after a crash — with checkStatus(eventId). See event_id & Reconciliation for the full persistence contract.
05

Tear-down

The SDK does not require an explicit close() / shutdown() call. At process exit:

✅
The worker pool is dropped and joined automatically.
✅
On Android, per-call HandlerThread instances have already been quit in their own finally block.
✅
No persistent file handles, network sockets, or listening ports to leak.
🔌
Explicit disconnect for long-running services

If you need to release transport resources before process exit — e.g. a long-running service doing dynamic re-config — call disconnect() first, then drop your reference. After disconnect(), any transport-touching method raises SdkError.NotConnected until you call connect() again (or App-to-App's per-call bind takes over).

06

Common mistakes

If you hit one of these symptoms, here is the likely cause and fix:

Symptom Cause & fix
IllegalStateException: Pine Labs SDK called from main thread You called cancel, checkStatus, connect, ping, etc. on the Android UI thread. Move the call to Dispatchers.IO or an Executor.
OperationInProgress(active_event_id = …) on the very next call You did not disable the trigger control in onStarted. The user tapped "Charge" twice before the first transaction settled.
Callback never fires Your TransactionListener threw an unhandled exception inside onStarted or onSuccess. The SDK delivered the callback, but your code threw and swallowed it. Add a top-level try/catch and log.
onSuccess fires but UI does nothing Your callback ran on the SDK worker thread; touching a TextView directly fails silently. Marshal to the UI thread with runOnUiThread { … }.
checkStatus returns Unknown for an ID you definitely allocated You constructed two SDK instances in the same process. The ID belongs to the other instance's registry. Consolidate to one instance (§01).
Cloud cancel hangs 30 s then Timeout One of: (a) called cancel on the UI thread on Android, (b) forgot cancelOptions = CancelOptions.Cloud { amount } and the server rejected it, (c) cloudConnectTimeoutMs is set too high. Check logs for the specific sub-reason.
