Pine Labs SDK

A single, language-idiomatic library that drives a Pinelabs payment terminal from your POS, restaurant, retail, or unattended-checkout software. One API. Every transport. Rust-core reliability.

entry point PineBillingSdk.doTransaction(request, listener)
01

What the SDK is

From your code's point of view, you call one method. The SDK handles everything beneath it:

🔀
Transport selection
Chooses the right transport for your environment — App‑to‑App, Cloud REST, PADController, or TCP — so you never write transport-specific code.
📦
Serialisation & framing
Serializes your request into the on-the-wire format the terminal understands. You never see CSV, JSON, framing bytes, or protocol details.
⚙️
Transport ceremony
Handles bind / connect / send / wait-for-reply / unbind — every per-transport lifecycle step that previously had to live in your app.
✅
Normalized result
Surfaces a typed TransactionResult on success, or a typed SdkError variant on failure — the same shape across all transports.
⚙️   Built in Rust — wrapped with UniFFI — same contract, same names, same validation rules in every language binding.

One distribution channel only

All artifacts are published exclusively on the Pinelabs Developer Portal (download) and via the Pinelabs MCP getSDK tool. The SDK is not on Maven Central, CocoaPods.org, PyPI, npmjs.com, or any private registry. Each artifact ships as a zip containing the binary plus a copy of these docs.

04

What is not in v1

Deliberately out of scope. Do not design your integration around these. They will ship — if at all — in a later phase:

  • ✗ Direct-from-SDK Bluetooth scanning / pairing. Bluetooth terminals are reached through PADController only. The SDK does not expose a Bluetooth radio API.
  • ✗ Direct USB enumeration on platforms other than Android. PADController is the supported path.
  • ✗ A typed enum for "Pinelabs cloud allowed-payment-mode". The Cloud transport accepts an opaque allowed_payment_mode string (commonly "10") until the typed mapping is published.
  • ✗ Wire-format APIs. CSV, JSON, framing — none are exposed in any binding. They are an internal implementation detail. Use the typed TransportOptions::* variants on the request for transport-specific extras.
  • ✗ Public package registries. Maven Central / CocoaPods.org / PyPI / npmjs.com publication. Use the Developer Portal or the MCP getSDK tool.

End-to-end flow at a glance

The same flow applies to testPrint, cancel, checkStatus, discoverTerminals, and every other transport-touching method.

💻
Your app — UI thread
call sdk.doTransaction(request, listener) — returns immediately, no blocking I/O on your thread.
▼
dispatches to worker thread
⚙️
PineBillingSdk — Rust core via UniFFI
1Validates the request (throws InvalidInput if bad)
2Allocates a UUIDv4 event_id for this operation
3Fires listener.onStarted(event_id) on worker thread — persist now
4Routes to the active transport
▼
selects active transport
🔀

App‑to‑App Cloud REST PADController TCP
On-the-wire format: CSV · JSON · framed bytes · HTTPS — all internal, never exposed to your code.

💳
