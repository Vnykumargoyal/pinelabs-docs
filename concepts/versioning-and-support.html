Versioning, support & compatibility

This page is the contract between the Pinelabs SDK and your code across releases. It defines the SemVer policy, minimum platform requirements, backwards-compatibility guarantees, deprecation rules, and end-of-life schedule.

⚠️
v1.0.0 is not yet tagged

Releases are currently tagged v0.x.y and may break compatibly within the v0 line at MINOR bumps to absorb late-breaking deltas. The v1.0.0 tag ships when the full §4 transport × language matrix (App-to-App / Cloud / PADController across Kotlin / Swift / Python / Node.js / C) is complete.

Action: Pin production integrations to an exact v0.x.y tag and re-validate on every bump.

01

SemVer policy

The SDK follows SemVer 2.0.0 — MAJOR.MINOR.PATCH. Each bump level carries a specific promise.

MAJOR Breaking changes
  • Removed or renamed public types or methods
  • Narrowed input validation rules
  • Removed metadata keys
  • Removed TransactionStatus variants
  • Behavioural change with no overload-and-deprecate path
MINOR Additive only
  • New optional fields or methods
  • New transports or TransportType entries
  • New PaymentMode / OperationState enum entries
  • New metadata keys
  • New SdkException variants (see exhaustiveness note)
PATCH No API surface change
  • Bug fixes and performance improvements
  • Internal refactors
  • Log-level tuning
  • Documentation updates
  • Dependency bumps without consumer-side change
💡
SdkException exhaustiveness

New exception variants are MINOR additions. Always include a default / wildcard branch when matching on SdkException so your code does not break on a MINOR bump:

Kotlin
when (e) {
    is SdkException.InvalidInput  -> handleInput(e)
    is SdkException.Timeout       -> handleTimeout(e)
    is SdkException.TransactionFailed -> handleFailed(e)
    // … other known variants …
    else -> log.warn("unknown SdkException variant: $e")
}

04

Backwards compatibility guarantees

The table below is the binding contract per bump level. The "Will not" list applies without a MAJOR; the "May do" lists are explicitly permitted.

🔒 Will not do without a MAJOR bump
  • Remove a public type, method, parameter, field, or enum variant
  • Remove an SdkException variant
  • Tighten validation rules (e.g. lower the metadata cap)
  • Change the meaning of an existing metadata key
  • Change the callback dispatch thread (always SDK-internal, always serialised)
  • Re-order callback emission (onStarted always before onSuccess / onFailure)
  • Change the synchronous-vs-async error delivery channel for an existing variant
➕ May do at MINOR (additive)
  • Add new fields, methods, enum variants, or metadata keys
  • Add new SdkException variants
  • Add new transports
  • Loosen validation rules
  • Promote a metadata key to a typed field (key remains populated for one MAJOR cycle)
🔧 May do at PATCH
  • Bug fixes and performance improvements
  • Log-level tuning and internal refactors
  • Dependency bumps without consumer-side change
05

Distribution & integrity

The SDK is distributed exclusively through official Pinelabs channels with cryptographic verification on every artifact.

🏛️
Official channels only
Every release artifact (AAR, xcframework, wheel, npm tarball, .tar.gz) is published exclusively on the Pinelabs Developer Portal and via the Pinelabs MCP getSDK tool.
🚫
Not on public registries
The SDK is not published to Maven Central, CocoaPods.org, PyPI, npmjs.com, or any private registry. Packages from those sources are unofficial and should not be used.
🔐
SHA-256 checksum + detached signature
Every artifact ships with a SHA-256 checksum and a detached signature, both downloadable from the same release page. Verify checksums before checking the artifact into your build system.
📦
Docs bundled in every release
Every artifact zip includes a copy of these docs at docs/site/index.html so you can browse offline.
📖
Per-language verification procedures

Step-by-step checksum verification instructions for each platform are documented in per-language/<lang>/setup.md inside the release zip.

06

Deprecation policy

Deprecated symbols follow a four-step lifecycle before removal. Removal only ever happens at the next MAJOR version.

1
Annotate the symbol
The symbol is marked with the language-idiomatic deprecation annotation: @Deprecated (Kotlin / Java), @available(*, deprecated) (Swift), DeprecationWarning (Python), @deprecated JSDoc (Node.js), __attribute__((deprecated)) (C).
2
Name the replacement
The deprecation message includes the name of the replacement symbol so IDEs and compilers surface the migration path inline.
3
Record in CHANGELOG
The CHANGELOG records the deprecation in the MINOR release that introduced it, so it appears in release notes.
4
Remove at the next MAJOR
The symbol is removed only at the next MAJOR version — never at a MINOR or PATCH — giving you a full MAJOR cycle to migrate.
07

Support channels

Use the right channel for your issue type — especially for security, where the public queue must not be used.

🐛
Bug reports & questions
Use the Pinelabs Developer Portal ticketing system. Include: the SDK version, your platform, the event_id (if any), and an excerpt of the SDK log at Debug level.
🔒
Security issues
Follow the process in SECURITY.md at the repository root. Do not file security issues in the public ticket queue — doing so may expose the vulnerability before a fix is available.
💡
Feature requests
Use the Developer Portal feedback channel. Feature work is scheduled into the next MINOR release.
08

End-of-life

The SDK supports the two most recent MAJOR lines at any time. When vN+2.0.0 ships, vN.x.y enters maintenance mode (security patches only) for 6 months, then reaches EOL. The CHANGELOG records EOL deadlines as soon as vN+1.0.0 ships.

Current
vN+1
Active development & bug fixes
Maintenance
vN
Security patches only · 6-month window after vN+2 ships
EOL
vN−1
No patches · upgrade required
📅
Deadlines are tracked in the CHANGELOG

As soon as vN+1.0.0 ships, the CHANGELOG entry records the exact EOL date for vN−1.x.y so you can plan migrations in advance.
