Error Handling

Every failure surfaces as a typed SdkError variant. This page is the reference for what each variant means, when it fires, how to recover, and whether you'll see it synchronously thrown or asynchronously delivered via a listener.

Synchronously thrown — no callback fires
Async via onFailure(error) on worker thread
01

The SdkError taxonomy

Two delivery channels exist for listener-based methods (doTransaction, testPrint, discoverTerminals):

⚡
Synchronously thrown — the call raises before any callback fires. Caused by input validation, invalid SDK state (NotConnected, OperationInProgress, NotSupported), or the transport refusing to start. No callback fires.
🔄
Async via listener — the call returns; later onFailure(error) fires on the worker thread. Covers all transport, terminal, timeout, and cancellation outcomes after the SDK has accepted the call.

InvalidInput(detail) Short, non-PII description ✓ —
TransportUnavailable(detail) Diagnostic (e.g. "MasterApp not installed") ✓ bind step ✓ mid-flight
Timeout(detail) Stage name — ✓
TransactionFailed(detail, terminalResponseCode?) Raw acquirer/terminal message + code — ✓
Internal(detail) Diagnostic; SDK-side defect ✓ or ✓ ✓
NotConnected() — ✓ —
OperationInProgress(activeEventId) Id of the in-flight op ✓ —
Cancelled() — — ✓
TransportError(detail) Mid-flight transport failure — ✓
NotSupported(detail) Method name (do not parse) ✓ —
02

Variant-by-variant reference

InvalidInput (detail)
Sync Not retryable

Fires when caller-supplied input fails validation. Always synchronous; never via a listener.

Common triggers:

  • amount <= 0 or amount > 99_999_999
  • billingRefNo is blank after trim
  • currency (when set) is not exactly 3 ASCII uppercase letters
  • originalEventId is missing for Refund / Void / Capture, or supplied for Sale / PreAuth / BalanceInquiry
  • referenceId longer than 64 chars
  • metadata has more than 10 entries, or any value longer than 256 chars
  • transportOptions variant does not match the active transport
  • On Cloud, transportOptions is null (the transactionNumber / sequenceNumber keys are mandatory)
  • cancel / checkStatus options mismatch the active transport
Recovery Fix the caller's input. Treat as a programmer-side bug; never retry blindly.
TransportUnavailable (detail)
Sync Async Retryable (backoff)

Fires when the active transport could not be reached or bound at the start of a call (synchronous), or the transport reports the link is gone (asynchronous, mid-flight).

Common triggers:

  • App-to-App: MasterApp not installed, bindService failure, service died mid-flight
  • Cloud: TCP-connect failure to cloudBaseUrl, DNS failure
  • PADController: local PADController service not running, named pipe unreachable
Recovery Surface "The terminal is not available" to the user. Retry with backoff. For App-to-App, prompt the user to install MasterApp from the Pinelabs Developer Portal.
Timeout (detail)
Async only Conditional

Fires when the SDK awaited a reply for longer than the configured timeout. Always asynchronous via the listener.

detail names the stage: service-bind, messenger-reply, http-read, tcp-read, pad-frame-reply, etc.

⚠️

Ambiguous commit state

The request may or may not have committed at the terminal. Do not silently retry a charge-issuing operation (Sale, PreAuth, Capture) without a checkStatus confirmation that the original did not commit.

TransactionFailed (detail, terminalResponseCode?)
Async only Not retryable

Fires when the terminal returned a non-success response. Always asynchronous via the listener.

  • detail — raw acquirer/terminal message. Pass through to logs and diagnostic UI verbatim.
  • terminalResponseCode — raw acquirer code (e.g. "05", "51", "1023"). The SDK does not classify these into retryable buckets.
Recovery Show the terminal's message to the cashier; let them decide whether to retry, accept cash, or ask for another card.
Internal (detail)
Sync Async Not retryable

Fires when the SDK or its bindings hit a defect — malformed reply, panic in the Rust core, encoding error.

Recovery Report to Pinelabs SDK support with the detail, your event_id, and a log capture. Do not silently retry — this is a bug, not a transient.
NotConnected ()
Sync Retryable (after connect)

Fires when a transport that requires a separate connect step (PADController, TCP) is in a disconnected state. Always synchronous.

Recovery Call connect(terminal) first, then retry.
OperationInProgress (activeEventId)
Sync Not retryable

Fires when the caller invoked a listener-based method while another op is in flight on the same SDK instance. Always synchronous.

📝

This is a UI bug

The trigger control was not disabled on onStarted. The SDK refuses to silently queue concurrent ops to avoid double-charge bugs. Disable the trigger control and wait for the in-flight op to complete.

Cancelled ()
Async only Caller-driven

Fires when the caller successfully cancelled an in-flight op that did not commit at the terminal. Asynchronous via the listener.

Recovery Show "Cancelled" to the user. Do not auto-retry. Re-issue with a new event_id if the user wants to try again.
TransportError (detail)
Async only Conditional

Fires when the underlying transport failed mid-stream — socket reset, IPC linkToDeath, HTTP body truncated. Asynchronous via the listener.

Recovery Treat like Timeout — the request may or may not have committed. Do not silently retry charge-issuing operations without a checkStatus.
NotSupported (detail)
Sync Not retryable

Fires when the active transport does not implement the called capability (e.g. cancel on App-to-App, discoverTerminals on Cloud). Always synchronous.

💡

detail is a human-readable diagnostic naming the method. Do not parse it programmatically — its casing and wording vary by language binding (e.g. Kotlin: "checkStatus is not supported on AppToApp"; Python: "check_status is not supported on AppToApp"). Use setTransport to switch to a transport that supports the capability, or guard the call with the capability matrix.

03

Retryability cheat sheet

Error Variant Retryable? Reason / Action
InvalidInput No Programmer error — fix the input, never retry blindly
TransportUnavailable Yes Transient link issue — retry with exponential backoff
Timeout Maybe Call may have committed — verify with checkStatus before retrying a charge
TransactionFailed No Card declined by acquirer — surface message to cashier
Internal No SDK defect — report with detail and event_id to Pinelabs support
NotConnected Yes Call connect(terminal) first, then retry
OperationInProgress No UI bug — disable trigger on onStarted; wait for in-flight op
Cancelled Caller-driven Re-issue with a fresh event_id if the user wants to retry
TransportError Maybe Treat like Timeout — verify commit state before retrying
NotSupported No Wrong transport — switch with setTransport or check the capability matrix
04

Top-level try/catch pattern

Catch synchronous errors at the call site, and handle asynchronous errors inside your listener. Never mix the two.

Kotlin 
fun chargeOrder(orderId: String, amountPaise: ULong) {
    // ── Sync error: SDK must be initialised before use ───────────
    try {
        sdk.doTransaction(buildRequest(orderId, amountPaise), uiListener)
    } catch (e: SdkException.NotInitialized) {
        log.error("Call PinelabsSdk.init() before doTransaction()")
    } catch (e: SdkException.InvalidInput) {
        log.error("Programmer error: ${e.detail}")
    } catch (e: SdkException) {
        toast("Could not start: $e")
    }
}

// ── Async errors: handle inside the listener ─────────────────
val uiListener = object : TransactionListener {
    override fun onStarted(eventId: String) {
        runOnUiThread { disablePayButton() }
    }
    override fun onSuccess(result: TransactionResult) {
        runOnUiThread { showSuccess(result) }
    }
    override fun onFailure(error: SdkException) {
        runOnUiThread {
            when (error) {
                is SdkException.TransactionFailed    -> showDecline(error.detail, error.terminalResponseCode)
                is SdkException.Timeout              -> showAmbiguous()   // check status!
                is SdkException.TransportUnavailable -> showAmbiguous()   // check status!
                is SdkException.InvalidInput         -> log.error("Programmer error: ${error.detail}")
                is SdkException.Internal             -> log.error("SDK error: ${error.detail}")
                is SdkException.NotInitialized       -> log.error("SDK not initialised")
            }
        }
    }
}

func chargeOrder(orderId: String, amountPaise: Int64) {
    // ── Sync errors: catch here ──────────────────────────────
    do {
        try sdk.doTransaction(buildRequest(orderId: orderId, amount: amountPaise),
                              listener: uiListener)
    } catch let e as SdkError.OperationInProgress {
        showToast("Already processing: \(e.activeEventId)")
    } catch let e as SdkError.InvalidInput {
        logger.error("Programmer error: \(e.detail)")
    } catch {
        showToast("Could not start: \(error)")
    }
}

// ── Async errors: handle inside the listener ─────────────────
class UiListener: TransactionListener {
    func onStarted(eventId: String) {
        DispatchQueue.main.async { self.disablePayButton() }  // prevent OperationInProgress
    }
    func onSuccess(result: TransactionResult) {
        DispatchQueue.main.async { self.showSuccess(result) }
    }
    func onFailure(error: SdkError) {
        DispatchQueue.main.async {
            switch error {
            case .transactionFailed(let detail, _): self.showDecline(detail)
            case .timeout:                          self.showAmbiguous()   // check status!
            case .transportError:                   self.showAmbiguous()   // check status!
            case .cancelled:                        self.showCancelled()
            case .transportUnavailable(let detail): self.showTerminalGone(detail)
            default:                                self.showGenericError(error)
            }
        }
    }
}

Node.js 
function chargeOrder(orderId, amountPaise) {
    // ── Sync errors: catch here ──────────────────────────────
    try {
        sdk.doTransaction(buildRequest(orderId, amountPaise), {
            // ── Async errors: handle inside the listener ─────
            onStarted(eventId) {
                disablePayButton();      // prevent OperationInProgress
            },
            onSuccess(result) {
                showSuccess(result);
            },
            onFailure(error) {
                if (error instanceof SdkError.TransactionFailed) {
                    showDecline(error.detail);
                } else if (error instanceof SdkError.Timeout ||
                           error instanceof SdkError.TransportError) {
                    showAmbiguous();     // check status!
                } else if (error instanceof SdkError.Cancelled) {
                    showCancelled();
                } else if (error instanceof SdkError.TransportUnavailable) {
                    showTerminalGone(error.detail);
                } else {
                    showGenericError(error);
                }
            }
        });
    } catch (e) {
        if (e instanceof SdkError.OperationInProgress) {
            showToast(`Already processing: ${e.activeEventId}`);
        } else if (e instanceof SdkError.InvalidInput) {
            console.error(`Programmer error: ${e.detail}`);
        } else {
            showToast(`Could not start: ${e}`);
        }
    }
}
C 
/* ── Async callbacks (wired at startup) ─────────────────────── */
static void on_started(const char *event_id, void *ctx) {
    disable_pay_button();              /* prevent OperationInProgress */
}

static void on_success(const pine_billing_result_t *result, void *ctx) {
    show_success(result);
}

static void on_failure(const pine_billing_error_t *err, void *ctx) {
    switch (err->variant) {
    case PINE_BILLING_ERR_TRANSACTION_FAILED:
        show_decline(err->detail);     break;
    case PINE_BILLING_ERR_TIMEOUT:
    case PINE_BILLING_ERR_TRANSPORT_ERROR:
        show_ambiguous();              break; /* check status! */
    case PINE_BILLING_ERR_CANCELLED:
        show_cancelled();              break;
    case PINE_BILLING_ERR_TRANSPORT_UNAVAILABLE:
        show_terminal_gone(err->detail); break;
    default:
        show_generic_error(err);       break;
    }
}

static const pine_billing_callbacks_t ui_callbacks = {
    .on_started = on_started,
    .on_success = on_success,
    .on_failure = on_failure,
};

/* ── Call site — sync errors: check return status ───────────── */
void charge_order(const char *order_id, int64_t amount_paise) {
    pine_billing_request_t *req = build_request(order_id, amount_paise);
    pine_billing_error_t    err = {0};

    pine_billing_status_t s =
        pine_billing_do_transaction(sdk, req, &ui_callbacks, &err);

    if (s != PINE_BILLING_OK) {
        switch (err.variant) {
        case PINE_BILLING_ERR_OPERATION_IN_PROGRESS:
            show_toast("Already processing"); break;
        case PINE_BILLING_ERR_INVALID_INPUT:
            log_error("Programmer error: %s", err.detail); break;
        default:
            show_toast("Could not start");
        }
        pine_billing_error_free(&err);
    }
    pine_billing_request_free(req);
}
